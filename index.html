I

What React Three Fiber Actually Is

Before anything technical, a short story. Three.js is a beautiful, powerful library — but writing it imperatively means saying, step by step: create this mesh, add it to the scene, set its position, update it every frame. React Three Fiber (R3F) lets you describe the same scene the way you describe a React UI: as a tree of components, re-rendered when state changes. The imperative work still happens — R3F does it for you.

Mental Model

If React renders components into the DOM, R3F renders components into a Three.js scene graph. The JSX tag <mesh> becomes an actual THREE.Mesh. Props become method calls. Unmounting disposes. You already know the rules — only the target changed.

Three.js in one paragraph

Three.js wraps WebGL so you can talk about objects instead of triangles. A scene contains objects. Objects have geometry (shape) and material (surface). A camera defines a point of view. A renderer draws the whole thing to a canvas each frame. R3F does not hide Three.js — it gives you declarative access to every bit of it.

Fig. 1.1 — R3F sits between your declarative code and the imperative Three.js world.
Why It Matters

Declarative 3D is not just stylistic. It means state drives the scene. Change a prop and the scene changes — no manual cleanup, no forgotten listeners, no stale references. For a portfolio that must respond to user choices (floor plan, lighting mode, selected room), this is decisive.

Your first scene

import { Canvas } from '@react-three/fiber'

export default function App() {
  return (
    <Canvas camera={{ position: [3, 3, 5] }}>
      <ambientLight intensity={0.4} />
      <directionalLight position={[5, 5, 5]} />
      <mesh>
        <boxGeometry args={[1, 1, 1]} />
        <meshStandardMaterial color="#c9a866" />
      </mesh>
    </Canvas>
  )
}

Read it as a sentence. A canvas, with a camera placed in space, containing soft ambient light and a strong directional light, and a single mesh whose shape is a box and whose surface is a muted gold. There is no new THREE.Mesh, no scene.add. The JSX is the scene description.

Common Mistake

Treating R3F like the DOM. A <mesh> is not a <div>. It cannot contain text nodes or arbitrary HTML. It contains geometry, material, and other 3D children. Mixing the two worlds is the first source of confusion — and the first source of clarity when you stop.

Exercise 1.1

In a fresh Vite project, install three and @react-three/fiber. Render a single cube. Change its color. Change its position by editing the position prop on the mesh. Notice: you never touched Three.js directly.

Real Estate Portfolio Connection

Your entire portfolio will live inside a single <Canvas>. Every room, every hotspot, every camera angle is just a component. The discipline you build in this chapter — thinking of a scene as a tree — is the same discipline that will let you drop a new property into the portfolio without touching a single line of rendering code.

II

The Canvas and the Scene Graph

The <Canvas> component is the entire world in a box. It creates a renderer, a scene, and a default camera, then hands you a React tree to populate it. Everything you build in this book lives inside it.

What Canvas does for you

A plain Three.js app requires you to manually set up a WebGL renderer, attach it to a DOM element, create a scene, create a camera, handle resizing, and start a render loop. Canvas does all of that, and keeps doing it as the window changes. You think about the scene; Canvas thinks about the plumbing.

Fig. 2.1 — The scene graph: a tree rooted at the scene, where every node has a parent transform.
Key Idea

A scene graph is a tree of objects where each child inherits its parent's position, rotation, and scale. Move the parent and the children move with it. This single property is the secret engine of composition in 3D.

Configuring the Canvas

<Canvas
  shadows
  dpr={[1, 2]}
  camera={{ position: [4, 3, 6], fov: 45 }}
  gl={{ antialias: true, alpha: false }}
>
  { /* scene */ }
</Canvas>

Three flags worth knowing early: shadows enables shadow mapping, dpr clamps device pixel ratio (critical for retina displays and performance), and camera overrides the default camera. You will tune these as your portfolio matures.

Common Mistake

Putting the <Canvas> inside a container with no explicit height. The canvas fills its parent — if the parent is zero pixels tall, so is your scene. Always give the wrapper a defined height (e.g. 100vh).

TypeScript Tip

R3F ships its own types for JSX intrinsics. <mesh>, <group>, and friends are all typed. If autocomplete fails, import @react-three/fiber once at the top of your entry file — the module augments the JSX namespace on import.

Exercise 2.1

Wrap three meshes in a <group>. Move the group. Confirm all three move together. Rotate the group. Confirm the children orbit the group origin, not their own.

Chapter Task

Recap

Canvas is the world. The scene graph is a tree. Children inherit parent transforms. Always give Canvas a sized parent.

III

Meshes, Geometry, and Materials

Every visible thing in a 3D scene is a mesh. A mesh is made of two parts: geometry, which is the shape (vertices and faces), and material, which is how the surface responds to light. Learning to separate these two in your head is half of becoming fluent in 3D.

Fig. 3.1 — A mesh is geometry plus material. Swap either one and the mesh transforms.

Built-in geometries

R3F exposes every Three.js primitive as a JSX tag: boxGeometry, sphereGeometry, planeGeometry, cylinderGeometry, torusGeometry, and more. You will use these constantly for prototyping — a box for a wall, a plane for a floor, a cylinder for a column.

Standard material, in plain English

meshStandardMaterial is the default workhorse. It is physically based, meaning its behavior mimics how real surfaces respond to light. Three props matter most at the start: color (the base hue), roughness (0 = mirror, 1 = chalk), and metalness (0 = dielectric, 1 = metal). A good architectural wall: off-white color, roughness around 0.9, metalness 0.

<mesh position={[0, 1, 0]}>
  <boxGeometry args={[2, 2, 2]} />
  <meshStandardMaterial
    color="#f3ead4"
    roughness={0.9}
    metalness={0}
  />
</mesh>
Why It Matters

Real estate visualization lives and dies on material realism. A "correct" wall with wrong roughness reads as plastic. Learning to feel the difference between 0.6 and 0.9 roughness is a genuine professional skill.

Common Mistake

Using meshBasicMaterial and wondering why your lights do nothing. Basic materials ignore lighting entirely — they are flat color. Use them for UI helpers, not for surfaces you want to feel real.

Exercise 3.1

Make a row of five spheres with identical color but roughness values of 0, 0.25, 0.5, 0.75, and 1. Add a single directional light. Study the difference. Which looks most like polished marble? Which looks most like matte plaster?

Chapter Task

Blocky building

Recap

Mesh = geometry + material. Standard material is the default. Roughness and metalness define surface feel.

IV

Lights, Cameras, and Atmosphere

A scene without lighting thought is a scene that feels like cardboard. In 3D, lighting is not decoration — it is the mood. You are not choosing bulbs, you are choosing a feeling.

The four lights you will actually use

Ambient light adds uniform illumination everywhere; it prevents shadows from being pure black. Directional light simulates the sun — parallel rays from one direction, and the only light that casts efficient shadows. Point light radiates from a point, like a lamp. Spot light casts a cone, like a flashlight or a gallery accent.

Fig. 4.1 — The four lights, each with a different physical metaphor.
Mental Model

Think like a film director. You rarely want one light. You want a key (main, usually directional), a fill (softer, ambient or low point), and maybe a rim (a spot from behind). Two or three well-placed lights beat ten random ones.

Cameras

Two cameras matter: PerspectiveCamera (the default — objects farther away look smaller, as in real life) and OrthographicCamera (no perspective distortion — useful for architectural plans and isometric views). For a real estate walkthrough you almost always want perspective. For a floor plan overview, orthographic reads more technical and honest.

Real Estate Portfolio Connection

Plan three signature lighting moods for your portfolio: morning (warm directional from low angle, cool ambient), midday (white directional from high angle, neutral ambient), evening (orange-red directional, blue ambient, warm point lights inside). A single "time of day" toggle can swap between them.

Exercise 4.1

Chapter Task

Atmosphere pass

Recap

Ambient fills, directional acts as sun, point is a lamp, spot is an accent. Cameras are either perspective (realistic) or orthographic (technical).

V

Transforms and Coordinate Space

Every object in a 3D scene has three transforms: position, rotation, and scale. Understanding how these combine — and how they nest inside groups — is the difference between fighting the scene and flowing with it.

Fig. 5.1 — The right-handed coordinate system R3F uses. +Y is up, +Z comes toward you.

Position

Position is a three-number array [x, y, z]. R3F also accepts dash-prop shortcuts: position-x={2}. Rotation uses radians. Scale is either a single number (uniform) or an array.

<mesh
  position={[2, 1, 0]}
  rotation={[0, Math.PI / 4, 0]}
  scale={[1, 2, 1]}
/>
Common Mistake

Using degrees instead of radians. Three.js — and therefore R3F — always works in radians. Convert with THREE.MathUtils.degToRad(45) or memorize that Math.PI is 180°.

Local vs world space

A child's position is relative to its parent. If you put a mesh at [1, 0, 0] inside a group positioned at [5, 0, 0], the mesh ends up at world position [6, 0, 0]. This is why groups are so powerful: move the parent, and everything inside moves together without recalculation.

Fig. 5.2 — Local transforms inside a group add to the group's transform to produce world position.
Exercise 5.1

Place three small spheres inside a group at local positions [-1,0,0], [0,0,0], [1,0,0]. Rotate the group by Math.PI/2 around Y. Predict, then verify, the new world positions.

Chapter Task

Recap

Transforms are position, rotation (radians), scale. Children inherit parent transforms. Groups are the primary composition tool.

VI

Groups and Composition

In React you make small components and compose them. In R3F you do exactly the same — you just compose spatial components. The group is your composition atom.

Why components matter more in 3D

A real estate portfolio scene can easily contain thousands of objects. Without composition you will drown. With composition, you think in terms of <Building>, <Room>, <Furniture>, <Hotspot>. Each component owns its own geometry, materials, and internal structure. The parent never has to know.

function Column({ position }) {
  return (
    <group position={position}>
      <mesh position={[0, 1.5, 0]}>
        <cylinderGeometry args={[0.2, 0.2, 3]} />
        <meshStandardMaterial color="#ece0c4" roughness={0.9} />
      </mesh>
      <mesh position={[0, 3.05, 0]}>
        <boxGeometry args={[0.6, 0.1, 0.6]} />
        <meshStandardMaterial color="#d9c99a" />
      </mesh>
    </group>
  )
}
Key Idea

A reusable 3D component is just a React function that returns a group. Props become position, rotation, color, variant. Everything you learned about React component design applies.

Fig. 6.1 — A property decomposed into named, reusable components.
Exercise 6.1

Chapter Task

Recap

Groups enable spatial composition. Reusable 3D components are functions returning groups. React discipline equals scene discipline.

VII

Refs in 3D

React refs let you reach into a DOM element when the declarative model is not enough. In R3F, refs let you reach into a Three.js object. They are your escape hatch to imperative behavior — rotation, animation, direct manipulation — without leaving the React paradigm.

import { useRef } from 'react'
import { useFrame } from '@react-three/fiber'
import * as THREE from 'three'

function SpinningCube() {
  const ref = useRef<THREE.Mesh>(null!)
  useFrame((_, dt) => {
    if (ref.current) ref.current.rotation.y += dt
  })
  return (
    <mesh ref={ref}>
      <boxGeometry />
      <meshStandardMaterial color="#c9a866" />
    </mesh>
  )
}
Fig. 7.1 — A ref connects the React world to a live Three.js object, which you then mutate directly.
Why It Matters

Setting React state 60 times a second is slow and wasteful. Refs let you animate without re-rendering. This is the single most important performance pattern in R3F.

Common Mistake

Using useState for per-frame values (rotation, position during animation). Always use refs for continuous values. Reserve state for discrete changes a user can see — selection, mode, filters.

TypeScript Tip

Type your refs precisely: useRef<THREE.Mesh>(null!) for a mesh, useRef<THREE.Group>(null!) for a group. The null! pattern asserts that the ref will be attached before use.

Exercise 7.1

Create a floating sphere that gently bobs up and down using ref.current.position.y = Math.sin(t). Compare performance with a version using useState — the ref version should not re-render at all.

Chapter Task

Subtle building breathe

Give your building group a ref and, via useFrame, make it rotate one-tenth of a radian per second. We will replace this with real camera movement later, but for now it proves your ref plumbing works.

Recap

Refs are the imperative escape hatch. Mutate Three.js objects through refs for per-frame work. Reserve state for user-facing discrete changes.

VIII

The Frame Loop and Animation

R3F renders a frame roughly every 16 milliseconds. The hook useFrame lets you hook into that loop. Every animation, camera movement, shader update, or physics step happens here.

Fig. 8.1 — Each frame: run callbacks, update scene, draw, paint, repeat.

useFrame anatomy

useFrame((state, delta) => {
  // state.clock.getElapsedTime() — seconds since start
  // state.camera — the active camera
  // state.scene — the root
  // delta — seconds since last frame (use for frame-rate independent motion)
})
Key Idea

Always multiply motion by delta. ref.current.position.x += 0.01 is a bug waiting to happen — its speed depends on frame rate. ref.current.position.x += 1 * delta moves one unit per second, regardless of hardware.

Tweened transitions with react-spring

For discrete animations — a hotspot highlighting, a camera moving to a room — reach for @react-spring/three. It plays nicely with R3F and gives you physics-based, interruptible animation.

import { useSpring, a } from '@react-spring/three'

function GrowOnHover() {
  const [hovered, setHovered] = useState(false)
  const { scale } = useSpring({ scale: hovered ? 1.2 : 1 })
  return (
    <a.mesh
      scale={scale}
      onPointerOver={() => setHovered(true)}
      onPointerOut={() => setHovered(false)}
    >
      <sphereGeometry />
      <meshStandardMaterial color="#c9a866" />
    </a.mesh>
  )
}
Real Estate Portfolio Connection

Your portfolio will have two animation families: continuous (subtle idle rotation, ambient motion, time-of-day sun sweep — useFrame) and transitional (camera flies to room, hotspot scales on hover — react-spring). Choosing the right tool for each keeps your code calm.

Exercise 8.1

IX

Interaction, Events, and Hotspots

Interactive 3D is mostly about answering one question: what did the user just click on? R3F gives you DOM-like event handlers on every mesh — onClick, onPointerOver, onPointerOut, onPointerMove — and handles the ray-casting math behind the scenes.

Fig. 9.1 — A click fires a ray from the camera through the pointer. The first intersected mesh receives the event.

Event propagation

By default the event bubbles through every intersected mesh, nearest first. Call event.stopPropagation() to prevent further handlers. This matters when a hotspot sits on top of a wall — you want the hotspot to handle the click, not the wall behind it.

function Hotspot({ position, label, onOpen }) {
  const [hovered, setHovered] = useState(false)
  return (
    <mesh
      position={position}
      onClick={(e) => { e.stopPropagation(); onOpen(label) }}
      onPointerOver={(e) => { e.stopPropagation(); setHovered(true) }}
      onPointerOut={() => setHovered(false)}
    >
      <sphereGeometry args={[0.15]} />
      <meshStandardMaterial
        color={hovered ? '#ffffff' : '#c9a866'}
        emissive={hovered ? '#c9a866' : '#000000'}
      />
    </mesh>
  )
}
Mental Model

A hotspot is a small interactive mesh with a label and a callback. Your portfolio is not "a scene with features" — it is "a scene with a set of hotspots". That single reframe is how you keep interaction manageable.

UI overlay and 3D scene relationship

Not every UI element should live in 3D. HTML is still the right medium for text-heavy panels, property listings, descriptions. The trick is linking the two: a 3D click opens a 2D panel; a 2D button moves the 3D camera.

Fig. 9.2 — 3D and 2D communicate through shared React state, usually via a store or context.
Common Mistake

Forgetting that the cursor should change on hover. Set document.body.style.cursor from your onPointerOver and onPointerOut handlers. Users need the affordance.

Exercise 9.1

X

Controls and Navigation

Users need a way to look around. The @react-three/drei library ships pre-built controls: OrbitControls, MapControls, PointerLockControls, and more. You rarely write controls from scratch.

import { OrbitControls } from '@react-three/drei'

<Canvas>
  <OrbitControls
    enablePan={false}
    minDistance={4}
    maxDistance={20}
    maxPolarAngle={Math.PI / 2.1}
  />
  { /* scene */ }
</Canvas>
Mental Model

Controls are not magic — they are a small component that mutates the camera each frame based on pointer input. You can always eject and write your own; most of the time you should not need to.

Clamping matters

Unclamped OrbitControls let users drop the camera underground, spin upside down, or zoom inside walls. Every control choice in the snippet above is a polish decision: disabling pan keeps the subject centered, clamping polar angle prevents inverted views, clamping distance prevents "inside a wall" bugs.

Scripted camera movement

When a hotspot is clicked, you want the camera to glide to a new position rather than teleport. Combine controls with react-spring, or use drei's CameraControls which exposes setLookAt() with built-in interpolation.

Real Estate Portfolio Connection

Your portfolio will have two navigation modes: overview (OrbitControls, clamped) and walkthrough (scripted camera moves between predefined vantage points). A single state machine decides which mode is active.

Exercise 10.1

Add OrbitControls. Clamp so the camera cannot go below the ground plane. Set min and max distance so the viewer cannot lose the building.

Chapter Task

Clicking a hotspot flies the camera

Each hotspot gets a cameraTarget prop — a position and look-at pair. Clicking it interpolates the camera to that pose over one second. Use drei's CameraControls or react-spring on the camera ref.

Recap

Use drei's controls. Clamp everything. Mix free navigation with scripted camera moves for polish.

XI

Loading Models and Assets

Primitives get you far, but eventually you want real geometry — furniture, architectural details, decorative props. The standard format is glTF (.gltf or the binary .glb), and drei's useGLTF hook makes loading one almost trivial.

import { useGLTF } from '@react-three/drei'

function Sofa(props) {
  const { scene } = useGLTF('/models/sofa.glb')
  return <primitive object={scene} {...props} />
}
useGLTF.preload('/models/sofa.glb')
Fig. 11.1 — The path from file to visible object.

Suspense

useGLTF suspends until the model is ready. That means the calling component must live inside a <Suspense> boundary — typically the entire scene.

<Canvas>
  <Suspense fallback={null}>
    <Sofa position={[0, 0, 0]} />
  </Suspense>
</Canvas>
Common Mistake

Forgetting Suspense and getting a cryptic error. R3F is strict about this — every async asset must be inside a boundary.

Why It Matters

Real portfolios use real models. A couch from a glTF file carries shape, materials, UVs, and sometimes animations — everything you would spend weeks authoring by hand. Learn to trust loaded models as first-class citizens.

TypeScript Tip

For type-safe access to named nodes, run npx gltfjsx model.glb. It generates a typed component with named refs for every mesh in the file — perfect for swapping materials, wiring up hotspots, or hiding parts.

Exercise 11.1

Find a free .glb (Poly Haven, Sketchfab CC models). Place it in /public/models/. Load it with useGLTF. Scale and position it until it sits correctly in your scene.

Chapter Task

Recap

glTF is the standard format. useGLTF loads and caches. Everything async goes inside Suspense. gltfjsx gives typed access.

XII

Reusable Scene Systems and Architecture

At this point you have a lot of parts. A serious portfolio needs a system — a set of conventions that make adding the next property, the next room, the next hotspot, feel like filling in a form rather than writing new software.

Folder structure

src/
  scene/
    Canvas.tsx          // the root Canvas + providers
    Lighting.tsx        // three-light rig + time of day
    Controls.tsx        // OrbitControls + scripted moves
  components/
    Property.tsx        // high-level wrapper
    Building.tsx        // one building
    Room.tsx            // one room inside a building
    Hotspot.tsx         // interactive point
  data/
    properties.ts       // pure data — no JSX
  state/
    useAppStore.ts      // Zustand store
  ui/
    PropertyPanel.tsx   // HTML overlay
    TimeToggle.tsx      // HTML control
Key Idea

Separate data, scene components, HTML UI, and state. Each folder has one job. When something breaks you know where to look. When something grows you know where it belongs.

Fig. 12.1 — The three worlds of a R3F application, connected through a single state store.

Data-driven scenes

Instead of hardcoding buildings, describe them as data:

// data/properties.ts
export const properties = [
  {
    id: 'villa-ivory',
    name: 'Villa Ivory',
    position: [0, 0, 0],
    model: '/models/villa.glb',
    hotspots: [
      { id: 'door', position: [0, 1, 2], label: 'Entrance' },
      { id: 'pool', position: [3, 0, 1], label: 'Infinity pool' },
    ],
  },
]

Your <Property> component maps over this data and renders the right components. Adding a new property is a single object. This is the payoff of all the discipline.

Real Estate Portfolio Connection

This is the heart of your capstone. Everything — models, hotspots, camera targets, panel content — flows from a single array of property objects. The rest of your app is a renderer.

Exercise 12.1

Move your hotspot definitions out of JSX and into a data file. Have the component map over the array. Add a second property with different hotspot positions — without duplicating component code.

Chapter Task

Recap

Separate data, scene, UI, state. Describe properties as data. Make adding the next one trivial.

XIII

Debugging in Three Dimensions

3D bugs have a special flavor. The scene renders but looks wrong. An object is "invisible" but actually 10,000 units away. Lighting is off because a normal is flipped. The camera is inside a wall. Debugging 3D is about having tools that reveal the invisible.

Fig. 13.1 — A debugging loop that works for almost any 3D mystery.

The debugging toolkit

Helpers. drei ships <axesHelper>, <gridHelper>, <Stats>, and wrappers for light helpers. Drop them in during development and delete for production.

Leva. The leva package gives you a floating control panel for any value — light intensity, camera fov, material roughness. Tuning becomes fast because the feedback loop is immediate.

Print positions. When something is "invisible" the first question is always: where is it? Log ref.current.position and confirm it is where you think.

Check your units. 1 unit = 1 meter is a healthy convention. Loaded models sometimes come in at 100× scale. A scale prop of 0.01 often reveals a missing building.

Solo a component. If your full scene is broken, render just the suspect component on a fresh Canvas. Bugs stop hiding once you remove the crowd.

Common Mistake

Assuming "it's invisible" means "it's not rendering". 90% of the time it is rendering somewhere you cannot see — behind the camera, under the ground, or 500 meters away.

Exercise 13.1

Deliberately break your scene: set a model's position to [1000, 0, 0]. Then use axesHelper, gridHelper, and leva to diagnose and fix it without reading your source.

XIV

The Performance Mindset

3D performance is ruthless and measurable. Sixty frames per second is the target. Every millisecond matters. Fortunately the optimizations you will actually need are few, and they become reflex with practice.

Fig. 14.1 — The four knobs you will actually turn.

Rules of thumb

Clamp DPR. dpr={[1, 2]} on Canvas. Retina screens can easily be rendering 4× the pixels of a standard one; clamping is free performance.

Instance repeats. If you need a hundred identical trees, use <Instances> from drei. One draw call, one hundred trees.

Reuse geometry and materials. Create them once with useMemo and share across components. A fresh geometry per mount is a leak.

Refs over state for animation. Revisit chapter 7 — this is the single biggest lever.

Suspense boundaries for code-splitting. Lazy-load a heavy interior model only when the user enters that room.

Why It Matters

A portfolio that stutters on a mid-range laptop is a portfolio nobody scrolls. Performance is a feature that earns attention.

Exercise 14.1

Add a hundred small meshes individually. Observe the frame rate with <Stats>. Replace them with <Instances>. Observe again. The difference is the lesson.

Reference

Glossary

Canvas
The R3F root component. Creates renderer, scene, camera, and frame loop.
Scene graph
The tree of 3D objects. Children inherit parent transforms.
Mesh
A visible 3D object composed of geometry and material.
Geometry
The shape of a mesh — vertices and faces.
Material
How a surface responds to light — color, roughness, metalness, etc.
Ambient light
Uniform illumination from all directions; prevents black shadows.
Directional light
Parallel rays from one direction; simulates the sun; casts shadows.
Point light
Light radiating from a single point; simulates a bulb.
Spot light
Conical light; simulates a flashlight or gallery accent.
Perspective camera
Camera with realistic foreshortening; distant objects look smaller.
Orthographic camera
Camera without foreshortening; useful for plans and isometric views.
Radian
Angle unit used by Three.js. Math.PI equals 180°.
Transform
The combination of position, rotation, and scale.
Local space
Coordinates relative to a parent group.
World space
Absolute coordinates in the scene.
Ref
A React handle to a live Three.js object for imperative mutation.
useFrame
Hook that runs a callback every rendered frame.
Delta
Seconds elapsed since the previous frame; multiply motion by it.
react-spring
Physics-based animation library that plays well with R3F.
Raycasting
Math that turns a pointer position into an intersection with 3D objects.
Hotspot
A small interactive mesh representing a point of interest.
OrbitControls
Drei component allowing the user to rotate, pan, and zoom around a target.
glTF / glb
Standard file formats for 3D models. .glb is the binary variant.
useGLTF
Drei hook that loads and caches a glTF file.
Suspense
React boundary required when loading async assets in R3F.
gltfjsx
CLI that generates a typed React component from a glTF file.
Instances
Drei component for rendering many copies of one mesh in a single draw call.
DPR
Device pixel ratio; clamping it caps the number of pixels rendered.
Draw call
A single GPU command to draw a mesh; the main performance cost to watch.
Frustum culling
Automatic optimization: objects outside the camera's view are not drawn.
Leva
A floating UI panel for tweaking values live during development.
Drei
A companion library of helpful R3F components and hooks.
Zustand
A small state management library ideal for sharing state between 3D and HTML layers.
Environment map
An image used to light and reflect in a scene, giving materials their surroundings.
Shadow map
A texture storing depth from a light's view; used to calculate shadows.
