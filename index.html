Where Does an R3F Experience Actually Live?

Before writing a single line of Three.js, you need to know where your work will be deployed. An R3F experience does not exist in a vacuum — it lives inside another product. Most luxury real estate sites are built in Framer or Webflow, not in a custom React application. Your 3D experience is a component inside that larger system. Understanding the deployment pattern from day one changes how you size components, host assets, manage state, and handle mobile.

There are four patterns you will encounter in professional work. Each chapter in this book includes a note on how the technique being taught affects whichever pattern your project uses.

Pattern A — Framer Code Component (most common for design studios)

Your R3F Canvas lives inside a Framer code component. The designer places it on the canvas like any other Framer element — dragging, resizing, connecting CMS properties. The component is deployed as part of the Framer site. This is the fastest path to a published experience with no backend.

Best when: The site is Framer-built. The developer is comfortable with React and TypeScript. CMS content is managed in Framer.

Pattern B — Webflow Custom Code Embed (most common for CMS-driven sites)

Your R3F application is built as a self-contained JavaScript bundle, hosted on a CDN, and bootstrapped into a target <div> via Webflow's custom code embed block. The Webflow site manages all other content; the 3D experience is injected at runtime.

Best when: The site is Webflow-built. The client team needs Webflow's CMS and Editor workflow. The 3D experience is a single embedded section, not the whole site.

Pattern C — Iframe / Externally Hosted (maximum isolation)

The R3F application runs at its own URL (e.g., experience.aureliaresidences.com) and is embedded in the marketing site via <iframe>. The host site and the 3D experience communicate via window.postMessage.

Best when: The experience is highly complex (multiple views, deep linking). The same experience must embed on multiple platforms. Complete CSS and JS isolation is required.

Pattern D — Hybrid React Application (full control)

The entire site is a custom React application. The 3D experience lives at a dedicated route (/experience) and is lazy-loaded. React Router handles navigation between the marketing pages and the 3D view.

Best when: The client needs a fully custom application. There is no platform constraint. The developer controls the full stack.

What R3F Is and How It Works

What you will build: A simple three-dimensional scene inside a React component—a lit grey cube on a dark background, rendered with proper perspective, lights, and shadows. This will be your first canvas.

React Three Fiber: A Translation Layer

React Three Fiber (R3F) is a React renderer for Three.js. Think of it this way: React is a system for describing user interfaces using components and JSX. Normally, React renders DOM elements (divs, buttons, paragraphs) into the browser. R3F hijacks that same system and renders Three.js objects (meshes, lights, cameras, geometry, materials) into a WebGL canvas instead.

You are not writing Three.js. You are writing React. You describe a 3D scene as a tree of React components, and R3F reconciles that description into the low-level Three.js API. When your state changes, the scene updates. When a prop changes, the 3D object reflects it. This is the superpower: all the React tooling, patterns, and mental models you know apply to 3D.

The Core Mapping

  • Canvas = The Three.js scene, camera, and renderer, bundled into one React component
  • JSX elements like <mesh>, <geometry>, <material> = Three.js objects added to the scene
  • Props (position, rotation, scale, color) = Object properties, synchronized reactively
  • Custom hooks like useFrame, useRef = Access to the animation loop and low-level objects

Installation

Before writing any code, install the dependencies: 

npm install three @react-three/fiber @react-three/drei

three is the 3D engine. @react-three/fiber is the React renderer. @react-three/drei is a utility library with pre-built helpers (cameras, controls, loaders, HTML rendering, and more).

Your First Canvas: App.tsx

import React from 'react';
import { Canvas } from '@react-three/fiber';
import { OrbitControls } from '@react-three/drei';

export default function App() {
  return (
    <div style={{ width: '100%', height: '100vh' }}>
      <Canvas
        shadows
        camera={{ position: [5, 5, 5], fov: 50 }}
      >
        {/* Ambient light: soft, directionless illumination */}
        <ambientLight intensity={0.6} color="#d0d0d0" />

        {/* Directional light: simulates sun, casts shadows */}
        <directionalLight
          position={[10, 15, 5]}
          intensity={1.2}
          color="#fff5e0"
          castShadow
          shadow-camera-left={-20}
          shadow-camera-right={20}
          shadow-camera-top={20}
          shadow-camera-bottom={-20}
        />

        {/* The ground plane that receives shadows */}
        <mesh
          position={[0, -2, 0]}
          rotation={[-Math.PI / 2, 0, 0]}
          receiveShadow
        >
          <planeGeometry args={[30, 30]} />
          <meshStandardMaterial color="#e8e8e8" />
        </mesh>

        {/* The grey cube: geometry + material */}
        <mesh position={[0, 0, 0]} castShadow receiveShadow>
          <boxGeometry args={[2, 2, 2]} />
          <meshStandardMaterial
            color="#888888"
            roughness={0.6}
            metalness={0.2}
          />
        </mesh>

        {/* Orbit controls: let the user rotate the view with mouse */}
        <OrbitControls />
      </Canvas>
    </div>
  );
}

Line-by-Line Explanation

import { Canvas } from '@react-three/fiber' — This is the component that sets up the Three.js scene, camera, and renderer. Everything inside Canvas becomes part of the 3D world.

shadows — A prop on Canvas that enables shadow rendering. Without this, even if you set castShadow on lights and meshes, nothing shadows. Think of it as a global switch.

camera={{ position: [5, 5, 5], fov: 50 }} — Positions the camera at coordinates (5, 5, 5) in 3D space. FOV (field of view) is 50 degrees; lower FOV feels zoomed in. The coordinate system is [x, y, z] where y is up.

<ambientLight intensity={0.6} color="#d0d0d0" /> — A soft, directionless light that illuminates everything equally. This is not shadows-capable; it's just base brightness and mood. Intensity 0.6 is gentle (1.0 is full brightness).

<directionalLight position={[10, 15, 5]} ... castShadow ... /> — Like sunlight; it comes from a specific direction and can cast shadows. The shadow-camera-* props define a box around the light's view—anything outside this box won't cast shadows. For architectural scenes, you set this big enough to contain your model. The color #fff5e0 is warm (afternoon sun).

<mesh ... castShadow receiveShadow> — The mesh must opt-in to shadow casting and receiving. castShadow means "I throw a shadow"; receiveShadow means "shadows from other objects appear on me."

<boxGeometry args={[2, 2, 2]} /> — A cube, 2 units wide, tall, and deep. The args prop holds the constructor arguments. You can pass [width, height, depth, widthSegments, heightSegments, depthSegments]; we omit segments because the default (1 per side) is fine for a simple box.

<meshStandardMaterial color="#888888" roughness={0.6} metalness={0.2} /> — The surface of the cube. meshStandardMaterial is physically-based (it respects lights properly). roughness is 0.6, so it scatters light a bit (not glossy); metalness is 0.2, so it has a tiny bit of metallic quality (not fully matte). Together, these simulate a slightly weathered concrete or stone surface.

<OrbitControls /> — Attaches mouse controls to the camera. Click and drag to rotate, scroll to zoom.

🧠 Mental Model

You are not writing Three.js. You are describing a 3D scene in React JSX. R3F watches your component and translates every JSX element, prop, and update into the corresponding Three.js object. When you change color="#ff0000" to color="#00ff00", R3F updates the material's color behind the scenes. The React mental model applies: components, props, hooks, state. Only the output is 3D instead of DOM.

⚠ Common Mistake

Forgetting to set a height on the container div. If the Canvas parent is height: auto or has no explicit height, the canvas will be 0 pixels tall and you'll see nothing. Always wrap Canvas in a div with width: 100%; height: 100vh (or explicit pixel height).

Geometry and Materials

What you will build: A scene with four distinct 3D shapes, each using a different material to show how surface properties shape visual perception. You'll understand the difference between a shape (geometry) and its appearance (material).

Geometry vs. Material

Every 3D object in Three.js has two independent parts:

  • Geometry — The shape: vertices, faces, edges. A cube, sphere, cone, or custom mesh.
  • Material — The surface: color, texture, reflectivity, roughness. How light bounces off the shape.

You can apply the same material to different geometries, or different materials to the same geometry. They are decoupled. A rough grey material makes concrete. A smooth white material with high reflectivity makes glazed ceramic. The geometry says "I am a sphere"; the material says "I am wet marble."

Common Geometries

R3F wraps Three.js geometry classes. Here are the ones you will use most:

  • <boxGeometry args={[w, h, d, ws, hs, ds]} /> — Rectangular solid. Width, height, depth, then optional segment counts (default 1 each).
  • <sphereGeometry args={[radius, ws, hs]} /> — Ball. Radius, width segments, height segments. More segments = smoother but slower.
  • <cylinderGeometry args={[r1, r2, h, seg]} /> — Tube or cylinder. Top radius, bottom radius (set equal for cylinder), height, segments.
  • <planeGeometry args={[w, h, ws, hs]} /> — Flat quad. For ground planes, walls, or surfaces.
  • <torusGeometry args={[radius, tubeRadius, seg1, seg2]} /> — Donut. Outer radius, tube thickness, segments.

Materials: Three Types

meshBasicMaterial — Unlit. Ignores all lights. Good for solid backgrounds, debug visuals, or special effects. Fast.

meshStandardMaterial — Physically-based rendering. Respects lights, shadows, and environment. Roughness and metalness control surface feel. This is the workhorse for architectural scenes.

meshPhysicalMaterial — Enhanced PBR with extra properties: transmission (glass), iridescence, sheen, clearcoat. For premium surfaces like glass, varnish, or exotic finishes.

Material Properties for Architecture

On meshStandardMaterial and meshPhysicalMaterial:

  • color — RGB hex or string. #888888 is grey.
  • roughness — 0 to 1. 0 = mirror-like (polished steel), 1 = completely matte (rough concrete).
  • metalness — 0 to 1. 0 = non-metal (stone, plastic, ceramic), 1 = pure metal (brushed steel, polished aluminum).
  • transmission (physical only) — 0 to 1. Makes the material transparent like glass. Combine with low roughness for clear glass, high roughness for frosted glass.

Concrete wall: color="#c9c9c9" roughness={0.9} metalness={0} — Light grey, very rough, not metallic.

Clear glass panel: color="#ffffff" roughness={0} metalness={0} transmission={1} — White base, perfectly smooth, fully transparent.

Brushed steel railing: color="#d0d0d0" roughness={0.3} metalness={1} — Light grey, slightly rough (brushed, not polished), fully metallic.

Line-by-Line Explanation

The scene has a ground plane and four objects arranged left to right. Each demonstrates a material category:

Box (position={[-5, 0, 0]}) — Matte concrete. Roughness 0.85 makes it absorb light; metalness 0 ensures no reflection. Grey color #8b8b8b reads as aged concrete.

Lights and Shadows

What you will build: A moody, professionally lit architectural scene with soft shadows and warm evening light. You will master the three light types that do 90% of the work in architectural visualization.

Three Core Lights

ambientLight — Borderless, even illumination from all directions. It does not cast shadows. Think of it as the sky's ambient glow. Use it to set overall brightness and mood. Intensity 0.4–0.7 is typical; higher makes everything washed out.

directionalLight — Light from a specific direction, like the sun. It casts shadows. It has a "shadow camera" that defines which part of the scene can shadow. Essential for dramatic architectural mood.

pointLight — Light from a point in space, radiating outward in all directions, like a light bulb. Can cast shadows. Useful for interior lighting, lamps, accent lights.

Shadow Setup

Shadows in R3F require three things:

  1. shadows prop on Canvas — Enables shadow rendering globally.
  2. castShadow on lights — The light throws shadows.
  3. castShadow and receiveShadow on meshes — This mesh casts shadow and/or receives shadow from others.

Shadows are expensive (slow). Only enable them on objects and lights you truly need. A ground plane always receives shadow. Lights that matter (sun, key interior light) cast shadow. Accent lights usually do not.

Lighting for Mood: Color Temperature

Light color shapes how a scene feels:

  • #fff5e0 — Warm, soft (afternoon sun, golden hour). Inviting, premium.
  • #ffffff — Neutral daylight. Crisp, clear.
  • #d0e0ff — Cool, overcast (rainy day, cool north light). Moody, contemporary.
  • #ffb366 — Very warm (sunset). Dramatic.
  • #a0a0ff — Cool blue (evening, artificial). Cinematic.

For a premium residential property, warm afternoon light (#fff5e0) with cool blue shadows creates luxury mood. Pair it with a soft ambient light in cool tones to make shadows rich, not black.

Line-by-Line Explanation

ambientLight intensity={0.5} color="#d0e0ff" — A cool blue ambient glow sets base mood and ensures shadows are not pure black (they are dark blue).

The shadow-camera props define a box that contains the renderable shadow area. Too small and shadows cut off; too large and shadow detail is lost. shadow-mapSize-width={2048} is high resolution for sharp shadows.

pointLight position={[-8, 5, -10]} intensity={0.4} ... distance={40} — A subtle fill light from the opposite side. Not bright enough to fight the sun, but enough to lift shadows slightly and add dimensionality. The distance prop limits how far the light reaches.

meshStandardMaterial color="#a89968" roughness={0.75} metalness={0.05} — The main building is warm beige (like stucco or stone), rough (not glossy), and nearly non-metallic. It reads as a natural material.

The scene has five meshes: ground, main box, roof, glass cylinder, and accent sphere. Each casts shadow where appropriate, creating a layered, dimensional feel.

Common Lighting Pitfalls

Too bright: If everything looks washed out white, your ambient + directional + point lights add up to overexposure. Dial back ambient intensity to 0.3–0.4. Use directional intensity around 1.5. This is more realistic and premium-looking.

Shadows too dark: If shadows are pure black with no detail, your ambient light is too dark. Increase it or make it a cooler color so it fills shadows subtly.

Flat lighting: Using only one ambient light makes everything look like a studio render with no form. Always use a directional light to create shadows and direction.

🔑 Key Concept

Shadows are not just darkness; they are the space between light and form. Soft shadows with warm sun and cool shadows create dimension and mood. This is the difference between a 3D render that feels like a photograph and one that looks like video-game graphics.

⚠ Common Mistake

Enabling shadows on every mesh and light. Shadows are computationally expensive. Enable shadow-casting only on lights and objects that matter visually. The ground must receive shadow. A directional light (sun) must cast shadow. Accent lights and background objects usually do not. Also, shadow-camera bounds (left, right, top, bottom) that are too large blur shadow detail; bounds too small crop shadows. Test and iterate.

Transforms and the Scene Graph

What you will build: A stylized architectural structure made of multiple primitive shapes grouped and positioned together. You will learn to think in 3D world coordinates and how to build complex objects from simple parts.

Position, Rotation, Scale

Every 3D mesh in R3F has three core properties:

  • position={[x, y, z]} — Where the object sits in 3D space. Arrays use [x, y, z].
  • rotation={[x, y, z]} — Euler rotation in radians. Math.PI is 180 degrees; Math.PI / 2 is 90 degrees.
  • scale={[x, y, z]} — Size multiplier. 1 is normal; 2 is twice as big; 0.5 is half size. Can be a single number for uniform scale.

The coordinate system: Y is up (not Z). Positive X is right, positive Z comes toward you (right-handed). This is the Three.js standard.

The Scene Graph and Groups

A <group> is a container for other meshes. When you move a group, all its children move together. When you rotate a group, children rotate around that group's origin. This is hierarchical composition—the foundation of 3D modeling. 

<group position={[0, 0, 0]} rotation={[0, 0.5, 0]}>
  <mesh position={[2, 0, 0]}>
    <boxGeometry args={[1, 1, 1]} />
    <meshStandardMaterial color="#ff0000" />
  </mesh>
  <mesh position={[-2, 0, 0]}>
    <boxGeometry args={[1, 1, 1]} />
    <meshStandardMaterial color="#00ff00" />
  </mesh>
</group>

This group has two cubes, one at +2 on X (right), one at -2 on X (left). If the group rotates 0.5 radians around Y, both cubes rotate together around the group's center at [0, 0, 0].

Centering and Pivoting

By default, a mesh's position is its center. To pivot rotation around a point other than the mesh's center, use a group: 

<group position={[0, 0, 0]} rotation={[0, Math.PI / 4, 0]}>
  {/* Rotate around the group origin, not the mesh center */}
  <mesh position={[3, 0, 0]} />
</group>

The mesh orbits around the group's position because the group's rotation happens first, then the mesh's local position is applied.

Refs and the Animation Loop

What you will build: A 3D scene where geometry subtly rotates and transforms over time. You will master the animation loop—the heartbeat that makes 3D scenes feel alive.

useRef for 3D Objects

In React, useRef creates a reference to a DOM element or object that persists across renders without triggering a re-render when it changes. In R3F, you use useRef to grab a handle to the underlying Three.js object. 

const meshRef = useRef(null);

return (
  <mesh ref={meshRef}>
    <boxGeometry />
    <meshStandardMaterial />
  </mesh>
);

After the component mounts, meshRef.current is the actual Three.js Mesh object. You can read and write properties: meshRef.current.position.x = 5, meshRef.current.rotation.y += 0.01, etc. These changes do not trigger React re-renders; they are direct mutations on the Three.js object.

useFrame: The Animation Loop

useFrame is an R3F hook that runs a callback every animation frame (~60 times per second). It receives a state object with clock (elapsed time) and other utilities. 

import { useFrame } from '@react-three/fiber';

useFrame(({ clock }) => {
  // This runs every frame
  meshRef.current.rotation.y += 0.01;
});

Delta Time: Frame-Rate Independence

Never hardcode frame increments like += 0.01. That works at 60 fps but breaks at 30 fps or 120 fps. Instead, use delta time (elapsed time since last frame): 

useFrame(({ clock, delta }) => {
  // delta is time in seconds since last frame
  meshRef.current.rotation.y += delta * 0.5; // Rotates at 0.5 rad/sec
});

Now your animation runs at the same speed regardless of frame rate. This is the professional standard.

⏱ Performance Budget — useFrame Best Practices

Three rules that prevent the most common performance bugs in production R3F code:

1. Never create objects inside useFrame. new Vector3() inside a useFrame callback allocates a new object 60 times per second. JavaScript's garbage collector will visibly stutter the frame rate as it cleans these up. Create all vectors, quaternions, and matrices once with useRef outside the callback, then reuse them inside it.

2. Always use delta-based animation. mesh.rotation.y += 0.01 runs at different speeds on a 30fps phone and a 120fps desktop. Use useFrame((state, delta) => { mesh.rotation.y += 0.5 * delta; }) — speed × delta is frame-rate independent.

3. Pause when off-screen. When the 3D experience is scrolled out of view (in a scroll-driven layout) or the browser tab is in the background, the loop should not run. In Chapter 19, you will add an Intersection Observer that pauses the loop when the canvas is not visible.

What you will build: An interactive 3D scene where clicking on objects triggers actions and hovering changes appearance. You will learn raycasting—how R3F knows which 3D object your mouse is pointing at.

The R3F Event System

R3F meshes support mouse and pointer events just like DOM elements:

  • onClick — User clicks on the mesh.
  • onPointerEnter — Pointer (mouse/touch) enters the mesh's bounding box.
  • onPointerLeave — Pointer leaves.
  • onPointerMove — Pointer moves over the mesh.

Behind the scenes, R3F uses raycasting: it casts an invisible ray from the camera through your pointer position and checks which 3D geometry it hits first. If it hits a mesh with an event handler, the handler fires.

Event Object

Event handlers receive an event object: 

onClick={(e) => {
  e.stopPropagation(); // Stop event from bubbling to other meshes
  console.log('Hit position:', e.point); // Where the ray hit in world space
  console.log('Which object:', e.object); // The Three.js mesh
}}

Cursor Changes and Visual Feedback

Change the cursor to indicate interactivity: 

onPointerEnter={() => {
  document.body.style.cursor = 'pointer';
}}
onPointerLeave={() => {
  document.body.style.cursor = 'auto';
}}

🔑 Key Concept

Raycasting is the bridge between 2D input (mouse pointer) and 3D space (which mesh you clicked). R3F handles raycasting automatically. You just attach event handlers to meshes, and R3F tells you when they are hit.

⚠ Common Mistake

HTML Inside 3D Scenes

What you will build: A 3D scene where styled information panels float at 3D world positions, anchored to hotspots or architectural features. You will learn to blend DOM and 3D seamlessly.

The <Html> Component

Drei provides <Html>, a component that renders a DOM element anchored to a 3D world position. It is not a texture or overlay; it is real HTML, positioned and scaled in 3D space. 

import { Html } from '@react-three/drei';

<Html position={[0, 5, 0]}>
  <div style={{ background: 'white', padding: '10px', borderRadius: '4px' }}>
    Feature Name
  </div>
</Html>

The div renders at world position [0, 5, 0]. If the camera moves, the HTML moves with it in 3D space. It is fully interactive: you can click links, type in inputs, scroll.

Html Props

  • position={[x, y, z]} — 3D position where the HTML appears.
  • occlude — If set, the HTML is hidden behind 3D geometry. occlude="blending" makes it fade as it goes behind objects.
  • distanceFactor — Scales the HTML based on distance from camera. Useful for labels that should stay readable at any distance.
  • center — Centers the HTML element (by default, top-left corner is at position).
  • transform — If true, the HTML respects the parent group's transformations.

Camera Controls and Guided Movement

What you will build: An interactive 3D experience where buttons trigger smooth camera transitions to preset viewpoints. You will learn to animate the camera and give users guided tours of a property.

OrbitControls Recap

Drei's OrbitControls lets users rotate the view around a target point using the mouse. Props you can customize:

  • enableZoom={true} — Allow scroll to zoom (default true).
  • enablePan={true} — Allow middle-click to pan (default true).
  • minDistance={1} — Closest the user can zoom in.
  • maxDistance={100} — Farthest the user can zoom out.
  • maxPolarAngle={Math.PI} — Prevent rotating below the horizon (default Math.PI allows full sphere).
  • autoRotate={true} — Camera auto-orbits without user input.
  • autoRotateSpeed={2} — Speed of auto-rotation.

Programmatic Camera Animation

To move the camera to a specific location smoothly, you can reference the camera and lerp (linear interpolate) its position in useFrame: 

import { useThree } from '@react-three/fiber';

const { camera } = useThree();

useFrame(() => {
  // Lerp camera position toward target
  camera.position.lerp(targetPosition, 0.05);
});

What you will build: A 3D scene where a pre-made architectural model (a GLB file) replaces your procedurally-built geometry. You will learn to load external 3D assets, manage loading states, and optimize for web performance.

GLB vs GLTF: What is the Difference?

GLTF (Graphics Language Transmission Format) is an open standard for 3D models. It is exported by Blender, Maya, SketchUp, and other 3D tools. It comes in two flavors:

  • .gltf — Text-based JSON with separate .bin and image files. Human-readable but requires multiple files.
  • .glb — Binary, self-contained. One file, easier to serve on the web. Preferred for web.

For web use, always choose GLB.

Downloading and Preparing a Model

Free architectural models can be found on Sketchfab. For this book, download a simple modernist house model:

If you prefer Sketchfab, filter by Downloadable + CC license + glTF and confirm the file size before downloading. Always check the license tab before using a model in a commercial project.

// The loaded model component
function PropertyModel() {
  const { scene } = useGLTF('/models/villa.glb');

  // Clone the scene once and configure shadows inside the same memo
  // so the traversal does not re-run on every render.
  const clonedScene = React.useMemo(() => {
    const copy = scene.clone();
    copy.traverse((node) => {
      if (node.isMesh) {
        node.castShadow = true;
        node.receiveShadow = true;
      }
    });
    return copy;
  }, [scene]);

  return <primitive object={clonedScene} position={[0, 0, 0]} scale={1} />;
}

Line-by-Line Explanation

const { scene } = useGLTF('/models/villa.glb') — Load the model. The hook caches it, so if you load the same file twice, the second load is instant.

const clonedScene = React.useMemo(() => scene.clone(), [scene]) — Clone the loaded scene. This is important if you ever render multiple instances of the model; sharing the same scene object causes issues. useMemo ensures the clone only happens once.

clonedScene.traverse((node) => { ... }) — Walk every node in the model's hierarchy. For every mesh, enable shadow casting and receiving. Many models are not set up for shadows; this ensures they work in your lit scene.

<primitive object={clonedScene} ... /> — Render a raw Three.js object (the cloned scene) as an R3F element. primitive is R3F's escape hatch for Three.js objects that don't have React components.

<Suspense fallback={<ModelFallback />}> — While PropertyModel is loading, show a simple grey box. Once loaded, PropertyModel renders instead.

Model Optimization: DRACO Compression

Large models slow down loading. Three.js supports DRACO compression, a codec that reduces GLB file size by 70%+ without losing quality. Drei's useGLTF can decode Draco-compressed GLBs automatically — there is no <Draco /> component. You enable it by passing a Draco decoder path as the second argument: 

// useGLTF accepts a Draco decoder path as its second arg.
// Use the Google-hosted decoder, or copy the files into /public/draco/.
const { scene } = useGLTF(
  '/models/villa-draco.glb',
  'https://www.gstatic.com/draco/versioned/decoders/1.5.6/'
);

// Optional: pre-warm the cache so the model starts downloading early.
useGLTF.preload(
  '/models/villa-draco.glb',
  'https://www.gstatic.com/draco/versioned/decoders/1.5.6/'
);

You compress the source model once with a CLI tool such as gltf-transform optimize input.glb output.glb --compress draco, then load the compressed file as shown above. For premium real estate websites, this is essential: a model that takes 30 seconds to load loses users.

🔑 Key Concept

Loading external 3D models is how you scale from simple primitives to rich, detailed environments. A Blender model of Villa Mirador can have thousands of polygons, custom materials, and architectural detail that would take hours to build procedurally. useGLTF + Suspense is the pattern for integrating pre-made assets.

⚠ Common Mistake

Loading a 100MB model without compression. Users on slow connections will see a loading screen for minutes. Always check model file size; use DRACO compression or LOD (level-of-detail) techniques. For web, aim for models under 5MB compressed. Also, if a model comes with many separate files (textures, materials), make sure they are all placed in the correct folder or the model will have missing textures.

Scene State — React State and Zustand

What you will build: A global state management system for your 3D experience. Multiple UI components and 3D objects will read and write shared state (selected hotspot, camera target, active panel). You will learn when to use local React state and when to reach for Zustand.

Local React State: When to Use

Use local React state (useState) for ephemeral, component-specific data:

  • Is a hotspot hovered? (local state is fine)
  • Is a panel open or closed? (local state is fine)
  • What text is in a form input? (local state)

Local state is simpler and faster. Use it by default.

Global State: When to Reach for Zustand

Use global state when multiple unrelated components need to read and write the same data:

  • Which residence is selected? (the hotspot, the info panel, and the camera all need to know)
  • What is the current camera target? (the camera controller and the nav buttons both set it)
  • Is the experience in "tour mode" or "free explore" mode? (affects hotspot visibility, camera constraints, UI)

For these cases, use Zustand, a lightweight global state library for React.

⚠ Common Mistake

Putting everything in global state. If only one component needs a value, keep it local. Global state makes debugging harder and introduces unnecessary coupling. Be selective.

Camera Choreography and Shot Design

What You Will Build

A cinematic camera system that smoothly transitions between three named positions: an establishing wide shot of the Aurelia Residences, an approach shot at the entrance, and a close-up detail view of the terrace. You'll implement this as a reusable component that responds to UI controls and Zustand state.

The Core Principle: Film Language in Three Dimensions

Cinematographers use three types of shots to tell a story: the wide establishing shot shows context and scale, the approach shot brings the viewer closer with emotional intent, and the detail close-up focuses attention on a specific feature. Your job as a 3D developer is to translate this language into code.

A wide establishing shot uses a higher FOV (field of view) — typically 45–50° — to capture the building in context. An approach shot narrows to 35–40° and moves the camera closer, creating intimacy. A detail close-up uses 25–30° FOV, moves very close, and focuses on texture and materiality. This is not abstract theory. It's the difference between a scene that feels premium and one that feels like a tech demo.

Camera Placement: A Framework I Use

One framing rule I keep coming back to for architectural shots: position the camera at roughly 45° to a corner of the building, around eye level (≈1.7 meters), and far enough away to see both the structure and its surroundings. Pointing straight at a flat face tends to look static; angling toward a corner gives the eye depth to read. It's a starting point, not a law — but it's a useful default.

Implementing CameraRig.tsx

// src/experience/CameraRig.tsx
import { useEffect, useRef } from 'react';
import { useThree, useFrame } from '@react-three/fiber';
import { Vector3 } from 'three';
import { useExperienceStore } from '../store/experienceStore';

interface CameraPosition {
  name: string;
  position: Vector3;
  target: Vector3;
  fov: number;
}

export function CameraRig() {
  const { camera } = useThree();
  const cameraPositions = useRef<CameraPosition[]>([
    {
      name: 'overview',
      // Wide establishing shot: elevated, distant, sees full building
      position: new Vector3(25, 18, 35),
      target: new Vector3(0, 8, 0),
      fov: 45,
    },
    {
      name: 'entrance',
      // Approach shot: eye level at building entry
      position: new Vector3(15, 1.7, 12),
      target: new Vector3(0, 2, 0),
      fov: 38,
    },
    {
      name: 'terrace',
      // Detail close-up: intimate view of terrace materials
      position: new Vector3(8, 3, 8),
      target: new Vector3(0, 3.5, 0),
      fov: 28,
    },
  ]);

  const activeShotName = useExperienceStore((state) => state.activeShot);
  const desiredPosition = useRef(new Vector3(25, 18, 35));
  const desiredTarget = useRef(new Vector3(0, 8, 0));
  const currentTarget = useRef(new Vector3(0, 8, 0));

  // On shot change, store the new desired position/target/fov
  useEffect(() => {
    const shot = cameraPositions.current.find((s) => s.name === activeShotName);
    if (shot) {
      desiredPosition.current.copy(shot.position);
      desiredTarget.current.copy(shot.target);
      camera.fov = shot.fov;
      camera.updateProjectionMatrix();
    }
  }, [activeShotName, camera]);

  useFrame(() => {
    // Smooth lerp camera position toward desired (0.05 ≈ 20 frames to settle)
    camera.position.lerp(desiredPosition.current, 0.05);

    // Smooth lerp the look-at point itself, then aim the camera at it
    currentTarget.current.lerp(desiredTarget.current, 0.05);
    camera.lookAt(currentTarget.current);
  });

  return null; // This is a logic component, no visual output
}

Line-by-Line Explanation

Vector3 imports and interface: We define a simple interface for camera positions: name, world position, target point to look at, and FOV. Three Vector3 instances store position and target throughout the animation.

Three camera positions: "overview" places the camera high and distant (25, 18, 35) and points it at the building's center (0, 8, 0), with a 45° FOV showing the full structure. "entrance" moves to eye level (1.7m tall) and closer (15, 1.7, 12), narrowing FOV to 38° for intimacy. "terrace" is a tight close-up (8, 3, 8) with 28° FOV, focusing on detail.

useExperienceStore hook: We read activeShotName from Zustand. When it changes, useEffect updates the target position, target look-at, and FOV. The camera.updateProjectionMatrix() call is essential — FOV changes don't apply until the projection matrix is recalculated.

useFrame lerp logic: Every frame, we smoothly move the camera toward the target position using lerp with factor 0.05. Lower factors (e.g., 0.02) create slower, more cinematic motion. We also update the camera's lookAt to point toward the target. This creates smooth, professional camera movement without jarring snaps.

⚠ Common Mistake

Directly setting camera.position and camera.lookAt in state changes, then calling updateProjectionMatrix only once. If you jump the camera instantly, the scene feels rushed and cheap. Always lerp over multiple frames and update the look-at every frame for smooth, professional motion.

✦ Why It Matters

Premium architectural sites move with purpose and restraint. The camera is your narrator. A well-choreographed shot sequence tells the story of the building: first, here it is in its setting (overview); then, welcome, come closer (entrance); finally, look at the details that prove the quality (terrace). This sequence, coded correctly, costs nothing in performance but gains everything in perceived value.

FOV and Camera Height

Wide FOV (50°+) feels dramatic and environmental — good for establishing shots. Narrow FOV (25–30°) feels documentary and focused — good for details. Eye level (1.7m) is the default human perspective. Elevated cameras (15m+) feel like drone footage and show scale. Experiment with these parameters in your shot definitions and watch how they change the emotional tone of the scene.

🔑 Key Concept

The camera is not a tool to look at the model. It is a storytelling device. Every position and FOV choice carries meaning. An overview shot says "here is the context." An entrance shot says "you are welcome here." A detail shot says "the quality is in the craft." Code this intentionality.

Lighting Mood for Architecture and Luxury Spaces

What You Will Build

A lighting system that shifts the entire mood of the Aurelia Residences with a single prop: morning light (cool, high contrast), afternoon light (warm, balanced, premium standard), and evening light (deep amber, dramatic). You'll learn that lighting is not about visibility — it's about emotion and atmosphere.

Functional Lighting vs. Mood Lighting

Most 3D scenes light for function: "make sure everything is visible." Premium scenes light for mood: "make the user feel something." A functional scene looks like a CAD render. A mood-lit scene feels like a photograph in an architecture magazine. The difference is not complexity — it's intentionality.

Functional lighting uses a neutral ambient light (white, intensity 1.0) and maybe a directional light straight from above. Mood lighting uses color (warm or cool), direction (from a corner, never straight), and varying intensity to create narrative. Morning light comes from the side and is cool (bluish) with high contrast — it suggests freshness and clarity. Afternoon light is warm gold and soft — it is the default premium real estate aesthetic. Evening light is deep amber and dramatic — it suggests luxury and exclusivity.

Implementing LightingRig.tsx

Ambient light: We look for an existing ambient light by name. If it doesn't exist, we create one. We update its color and intensity based on the mood preset. Ambient light has no position — it lights uniformly from all directions.

🔑 Key Concept

A cool morning light makes the viewer feel alert and awakened. A warm afternoon light says "you can relax here." A deep amber evening light says "this is rare and special."

⚠ Common Mistake

Over-lighting the scene to make sure everything is visible. A 3D scene with every surface equally bright looks like a showroom or a video game, not a luxury property. Restraint is luxury. Use shadows. Let some areas be mysterious. The eye is drawn to light — so light strategically, not everywhere.

✦ Why It Matters

Professional architectural photographers spend hours on lighting. They understand that the same building looks different at different times of day, and they choose the time that best tells its story. You are coding that same intentionality. A single mood toggle on your site should make the entire experience feel like a different photograph. Motion Pacing, Restraint, and Editorial Transitions

What You Will Build

A scroll-driven camera animation system where the user scrolls the page and the camera smoothly moves through the Aurelia Residences from overview to entrance to terrace. You'll learn that motion without restraint feels anxious, but motion with intentional pacing and dwell (pausing) feels editorial and premium.

Restraint Is Luxury

Most 3D websites move the camera constantly. Premium sites move slowly and pause between movements. This is the principle of restraint applied to animation. When you scroll a luxury brand's website, you expect the experience to feel curated, not frantic. The camera should dwell at each major frame for a beat before moving. This makes the user feel guided, not rushed.

Implementing animateCamera Utility

First, we'll build a utility function that animates the camera from one position to another over time with easing.

// src/experience/animateCamera.ts
import { Vector3, Euler, Camera } from 'three';

export type EasingFunction = (t: number) => number;

// Easing functions
export const easing = {
  linear: (t: number) => t,
  easeInOutQuad: (t: number) => t < 0.5 ? 2 * t * t : 1 - Math.pow(-2 * t + 2, 2) / 2,
  easeInOutCubic: (t: number) =>
    t < 0.5 ? 4 * t * t * t : 1 - Math.pow(-2 * t + 2, 3) / 2,
};

export interface AnimateCameraOptions {
  fromPosition: Vector3;
  toPosition: Vector3;
  fromTarget: Vector3;
  toTarget: Vector3;
  duration: number; // milliseconds
  easing?: EasingFunction;
  onComplete?: () => void;
}

export class CameraAnimator {
  private startTime: number = 0;
  private isAnimating: boolean = false;
  private options: AnimateCameraOptions | null = null;
  private initialPosition: Vector3 = new Vector3();
  private initialTarget: Vector3 = new Vector3();

  start(camera: Camera, options: AnimateCameraOptions) {
    this.options = options;
    this.startTime = Date.now();
    this.isAnimating = true;
    this.initialPosition = options.fromPosition.clone();
    this.initialTarget = options.fromTarget.clone();
  }

  update(camera: Camera, deltaMs: number): boolean {
    if (!this.isAnimating || !this.options) return false;

    const elapsed = Date.now() - this.startTime;
    const progress = Math.min(elapsed / this.options.duration, 1);
    const easedProgress = (this.options.easing || easing.easeInOutQuad)(progress);

    // Lerp camera position and target based on eased progress
    camera.position.lerpVectors(
      this.options.fromPosition,
      this.options.toPosition,
      easedProgress
    );

    const currentTarget = this.options.fromTarget
      .clone()
      .lerp(this.options.toTarget, easedProgress);
    camera.lookAt(currentTarget);

    // Animation complete
    if (progress >= 1) {
      this.isAnimating = false;
      if (this.options.onComplete) {
        this.options.onComplete();
      }
      return false;
    }

    return true;
  }

  isRunning(): boolean {
    return this.isAnimating;
  }

  stop() {
    this.isAnimating = false;
    this.options = null;
  }
}

CameraAnimator Explanation

Easing functions: We provide three easing options. linear is the default — no easing, constant speed. easeInOutQuad starts slow, accelerates, then decelerates — feels smooth and organic. easeInOutCubic is similar but more dramatic. Always use easing; linear movement feels robotic.

AnimateCameraOptions: This object defines an animation: from and to positions, from and to targets (where the camera looks), duration in milliseconds, an optional easing function, and an optional callback when the animation finishes.

CameraAnimator class: This class manages the animation state. start() records the start time and initial positions. update() is called every frame and computes how far through the animation we are (progress 0–1), applies easing to that progress, and uses lerpVectors to smoothly move the camera position and lookAt toward the targets. When progress reaches 1, the animation ends and calls the onComplete callback if provided.

Scroll-Driven Animation with Dwell

Now we'll build a custom React hook that listens to scroll events and moves the camera along a predefined path.

// src/experience/useScrollCamera.ts
import { useEffect, useRef } from 'react';
import { useThree, useFrame } from '@react-three/fiber';
import { Vector3 } from 'three';
import { useExperienceStore } from '../store/experienceStore';

export interface ScrollCameraWaypoint {
  position: Vector3;
  target: Vector3;
  fov: number;
}

interface ScrollCameraOptions {
  waypoints: ScrollCameraWaypoint[];
  totalScrollHeight?: number; // Default: 3 * window.innerHeight
  enabled?: boolean;
}

export function useScrollCamera(options: ScrollCameraOptions) {
  const { camera } = useThree();
  const scrollProgress = useRef(0);

  const { waypoints, totalScrollHeight = window.innerHeight * 3, enabled = true } = options;

  useEffect(() => {
    if (!enabled) return;

    const handleScroll = () => {
      // Compute scroll progress as a number from 0 to 1
      const scrollTop = window.scrollY;
      const progress = Math.min(scrollTop / totalScrollHeight, 1);
      scrollProgress.current = progress;
    };

    window.addEventListener('scroll', handleScroll);
    return () => window.removeEventListener('scroll', handleScroll);
  }, [totalScrollHeight, enabled]);

  useFrame(() => {
    if (!enabled || waypoints.length === 0) return;

    const progress = scrollProgress.current;
    // Map scroll progress to waypoint index (0 to waypoints.length - 1)
    const waypointIndex = Math.min(
      Math.floor(progress * waypoints.length),
      waypoints.length - 1
    );

    const waypoint = waypoints[waypointIndex];

    // Smoothly move camera to waypoint position over 10 frames
    camera.position.lerp(waypoint.position, 0.1);

    // Smoothly update camera look-at
    camera.lookAt(waypoint.target);

    // Smoothly update FOV
    camera.fov += (waypoint.fov - camera.fov) * 0.05;
    camera.updateProjectionMatrix();
  });
}

useScrollCamera Explanation

Scroll event listener: We add a scroll event listener that updates scrollProgress.current based on the user's scroll position relative to totalScrollHeight. As the user scrolls from top to bottom, scrollProgress goes from 0 to 1.

useFrame mapping: Every frame, we compute which waypoint the user is currently "at" based on scrollProgress. We then smoothly lerp the camera position, target, and FOV toward that waypoint. The 0.1 lerp factor means the camera will smoothly catch up to the waypoint over about 10 frames.

Waypoints: Each waypoint has a position, target, and FOV. As the user scrolls, the camera interpolates between waypoints. If you have three waypoints (overview, entrance, terrace), the camera will smoothly transition from one to the next as the user scrolls through the page.

🧠 Mental Model

Think of scroll-driven camera animation like a film camera on a dolly track. The track is your waypoint path. As the user scrolls (plays the tape), the camera moves along the track. Fast scrolling means the camera moves faster; slow scrolling means it pauses. The dwell between movements is like a still frame in a film — the viewer takes time to absorb what they're seeing before the camera moves again.

🔑 Key Concept

Dwell is silence in animation. In film, directors use silence (and stillness) to build tension and allow the audience to absorb. In 3D web design, dwell is a camera pause at each key framing. Implement dwell by holding the camera at a waypoint for 1–2 seconds before moving. This makes the experience feel edited, not algorithmic.

⚠ Common Mistake

Constant motion without pause. If the camera never stops, the viewer never absorbs. Your scene becomes a demo, not an experience. Always include dwell time (even if just 500ms) at each key frame. This one detail elevates your scene from amateur to professional.

Environment, Atmosphere, and Reflection

What You Will Build

What Is HDRI and Why Premium Scenes Use It

An HDRI is a 360° photograph of a real space's lighting. It captures the actual light in a patio, a city, a sky, etc. When you use an HDRI as your scene's environment, three things happen: (1) the HDRI image becomes the background, (2) the HDRI provides realistic ambient lighting from all directions, and (3) reflective materials on your 3D models reflect the HDRI image, creating photorealism. A material with metalness 0.5 and roughness 0.2 will reflect the HDRI realistically — this is non-negotiable for premium real estate.

Sources for Free HDRI

Poly Haven (https://polyhaven.com/hdris): The industry standard. Free, high-quality HDRIs. For architectural scenes, download "Patio Compound" or "Apartment" (outdoor mood) or "Studio" (indoor mood). Download as .hdr format, 1k or 2k resolution.

Recommended HDRIs for real estate:

  • "Patio Compound" — warm, sunny outdoor patio setting. Great for ground-level terrace views.
  • "Apartment" — natural window light indoors. Use for interior residential views.
  • "Modern Office" — clean, bright, minimalist. Good for contemporary architecture.

import { AureliaModel } from './AureliaModel';
import { CameraRig } from './CameraRig';
import { LightingRig } from './LightingRig';
import { HotspotManager } from './HotspotManager';
import { useExperienceStore } from '../store/experienceStore';

export function PropertyExperience() {
  const mood = useExperienceStore((state) => state.mood);

  return (
    <Canvas
      shadows
      dpr={[1, 2]} // 1x on mobile, 2x on desktop (set by ResponsiveCanvas)
      camera={{ position: [25, 18, 35], fov: 45 }}
    >
      <Suspense fallback={null}>
        {/* Main 3D model */}
        <AureliaModel />

        {/* Cinematic camera system */}
        <CameraRig />

        {/* Environmental lighting and atmosphere */}
        <LightingRig mood={mood} />

        {/* HDRI environment: provides ambient lighting and reflections */}
        {/* Use background={true} to display HDRI as background, background={false} to keep React background */}
        <Environment
          files="/hdri/patio.hdr"
          background={false}
          intensity={0.8} // 0-1: how much the HDRI contributes to lighting
        />

        {/* Realistic ground shadows via ray-traced contact shadow */}
        <ContactShadows
          position={[0, -0.01, 0]} // just below ground plane
          opacity={0.4}
          scale={50}
          blur={2}
        />

        {/* Atmospheric fog for depth perception */}
        <fog attach="fog" args={['#e8e0d0', 50, 150]} />

        {/* Interactive hotspots */}
        <HotspotManager />
      </Suspense>
    </Canvas>
  );
}

Line-by-Line Explanation

Environment component: files="/hdri/patio.hdr" loads the HDRI image from your public folder. intensity={0.8} controls how much the HDRI contributes to ambient lighting (0 = no contribution, 1 = full contribution). background={false} means the HDRI will light the scene and provide reflections, but won't replace the canvas background — useful if you have a custom background color or a separate UI background.

ContactShadows: This drei component renders soft, realistic shadows on the ground where objects touch the surface. It's much more elegant than Three.js's default shadow mapping for ground contact. position={[0, -0.01, 0]} places the shadow plane just below ground (y=0). scale={50} sets the shadow's physical size. blur={2} softens the shadow edges.

fog component: This creates exponential fog that fades distant objects into the background color. args=['#e8e0d0', 50, 150] means the background color is cream, fog starts at 50 units away, and is completely opaque at 150 units. This creates atmospheric perspective — a sense that the scene extends beyond what you can see.

  mieCoefficient={0.005}
  mieDirectionalG={0.7}
/>

Sky is lighter-weight than HDRI (no file to download) but provides less photorealism. Use HDRI for luxury real estate; use Sky for simplified, stylized scenes.

🔑 Key Concept

HDRI is the foundation of photorealistic 3D web design. Without it, your scene looks like a CAD render. With it, your scene looks like a photograph. The key is the environment's effect on material reflection — a metallic or polished material should reflect the HDRI realistically. Always ensure your materials have metalness > 0 and roughness < 0.5 to showcase the HDRI's reflections.

⚠ Common Mistake

Using a default white or gray background without an HDRI. The scene instantly looks amateurish. Always load an HDRI, even a simple one. The difference in perceived quality is enormous and costs nearly nothing in performance (HDRI is lightweight because it's baked into the lighting calculations).

✦ Why It Matters

Premium architectural photography and visualization is about context. A building doesn't exist in a void — it exists in a world with light, air, and surroundings. HDRI brings that world into your 3D scene. It's the difference between showing a building and showing a building in a place. The user's brain recognizes this difference instantly, even if they don't consciously know why the HDRI version feels more "real."

What Makes 3D Feel Premium (Art Direction)

What You Will Build

Not a new feature, but a critical design framework. This chapter teaches you to recognize what separates a premium 3D experience from an amateur one. You'll audit your PropertyExperience.tsx against seven principles, and we'll show before-and-after comparisons of the same building rendered poorly vs. beautifully.

Seven Principles I Lean On

1. Restraint — Premium spaces move slowly. Lights dim intentionally. The camera pauses. Nothing is constantly animating or pulsing. A scene with constant motion feels frantic; a scene with deliberate stillness feels curated.

2. Material Quality — Cheap materials have uniform color (metalness 0, roughness 1). Premium materials respond to light realistically. Metalness 0.5–0.8 with roughness 0.1–0.4 feels like brushed aluminum or polished stone. Always texture your materials; plain color is death.

3. Lighting Mood — No pure white light. All lights have color (warm or cool). No scene should be equally bright everywhere. Shadows and subtle darkness are luxurious, not functional.

4. Camera Intentionality — Every camera position was chosen deliberately. No camera sits directly facing a flat surface. Angles of 30–45° to building corners create drama. Wide establishing shots (FOV 45–50°) frame context. Close-up detail shots (FOV 25–30°) emphasize craft.

5. Whitespace in 3D — Empty space is not wasted space. A scene with breathing room feels calm. A cluttered scene feels chaotic. If every inch of the frame is filled with detail, the viewer's eye doesn't know where to look. Premium design gives the eye a place to rest.

6. Typography Integration — Text on the page is part of the visual design. Font choices (serif or sans), color (always sufficient contrast), and placement (never centered or over the 3D; usually to the side or below) make or break the experience. Premium sites use high-quality typefaces (Cormorant Garamond for serif, Inter for body text) and intentional hierarchy.

7. Pacing — The experience has rhythm. Overview. Pause. Approach. Pause. Detail. This rhythm is set by the scroll speed, the camera dwell time, and the animation easing. Fast easing + no dwell = chaos. Slow easing + long dwell = tedium. The right balance feels natural, like a film.

The Bad Version (Common Mistakes):

Hotspots and Residence Storytelling

A hotspot is a small visual marker (usually a pulsing sphere or icon) at a specific world position in your 3D scene. When a user hovers over it, the cursor changes and text appears. When they click it, a detailed information panel opens. Hotspots transform passive viewing into active storytelling — the user is guided through the property feature by feature, learning about materials, views, functionality, and luxury.

Compare two approaches: (1) "Features list on the right sidebar in text" — the user reads a list. (2) "Walk to the feature in 3D, click a hotspot, see the detail" — the user experiences the feature in context. The second approach is immersive and memorable.

Implementing Hotspot.tsx

// src/experience/Hotspot.tsx
import { useRef, useState } from 'react';
import { useFrame } from '@react-three/fiber';
import { Mesh, Vector3 } from 'three';
import { Html } from '@react-three/drei';

interface HotspotProps {
  position: [number, number, number];
  label: string;
  description: string;
  onClick: () => void;
}

export function Hotspot({
  position,
  label,
  description,
  onClick,
}: HotspotProps) {
  const meshRef = useRef<Mesh>(null);
  const [isHovered, setIsHovered] = useState(false);

  // Pulsing animation: scale oscillates between 0.8 and 1.2
  useFrame(({ clock }) => {
    if (meshRef.current) {
      const pulse = Math.sin(clock.getElapsedTime() * 3) * 0.2 + 1; // 0.8–1.2
      meshRef.current.scale.setScalar(pulse);
    }
  });

  return (
    <group position={position}>
      {/* Invisible interactive sphere */}
      <mesh
        ref={meshRef}
        onClick={(e) => {
          e.stopPropagation(); // Prevent click from bubbling to Canvas
          onClick();
        }}
        onPointerEnter={() => {
          setIsHovered(true);
          document.body.style.cursor = 'pointer';
        }}
        onPointerLeave={() => {
          setIsHovered(false);
          document.body.style.cursor = 'auto';
        }}
      >
        <sphereGeometry args={[0.5, 32, 32]} />
        <meshStandardMaterial
          color={isHovered ? '#c9a866' : '#a8864b'} // gold: soft or normal
          emissive={isHovered ? '#a8864b' : '#000000'}
          metalness={0.6}
          roughness={0.3}
        />
      </mesh>

      {/* Tooltip label: appears on hover */}
      {isHovered && (
        <Html scale={0.08}>
          <div
            style={{
              position: 'absolute',
              backgroundColor: '#a8864b',
              color: '#fbf8f3',
              padding: '8px 12px',
              borderRadius: '4px',
              fontSize: '14px',
              fontFamily: 'Inter, sans-serif',
              whiteSpace: 'nowrap',
              pointerEvents: 'none',
              top: '-40px',
              left: '50%',
              transform: 'translateX(-50%)',
            }}
          >
            {label}
          </div>
        </Html>
      )}
    </group>
  );
}

Line-by-Line Explanation

Pulsing animation: In useFrame, we compute a pulse value using Math.sin. The sine function oscillates between -1 and 1; we map it to 0.8–1.2 by multiplying by 0.2 and adding 1. We apply this to the mesh scale, so the hotspot sphere gently expands and contracts. This draws the eye without being distracting.

Click handling: onClick stops event propagation (so the click doesn't reach the Canvas and trigger camera movement or other global events), then calls the parent's onClick callback.

Hover state: onPointerEnter sets isHovered to true and changes the cursor to pointer. onPointerLeave resets it. This gives the user clear feedback that the hotspot is interactive.

Material and color: The hotspot uses a metallic gold material (meshStandardMaterial with metalness 0.6, roughness 0.3) that reflects the HDRI. On hover, the color brightens and emissive increases, making the hotspot glow slightly.

Html tooltip: We use the Html component from drei to render a DOM element (the label) positioned above the hotspot in 3D space. The label only appears when isHovered is true. This is a key pattern: 3D geometry + HTML overlays = interactive storytelling.

Scroll-Driven Camera Movement

What You Will Build

A complete, production-ready scroll-driven camera system. As the user scrolls down the page, the camera smoothly moves from the overview shot to the entrance to the terrace detail. You'll implement the useScrollCamera hook (teased in Chapter 13), integrate it into PropertyExperience, and learn how to handle edge cases like pointer events and OrbitControls conflicts.

The Complete useScrollCamera Hook

// src/experience/useScrollCamera.ts
import { useEffect, useRef } from 'react';
import { useThree, useFrame } from '@react-three/fiber';
import { Vector3 } from 'three';

export interface ScrollCameraWaypoint {
  position: Vector3;
  target: Vector3;
  fov: number;
}

interface UseScrollCameraOptions {
  waypoints: ScrollCameraWaypoint[];
  totalScrollHeight?: number; // in pixels
  enabled?: boolean;
}

export function useScrollCamera(options: UseScrollCameraOptions) {
  const { camera } = useThree();
  const scrollProgress = useRef(0);
  const { waypoints, totalScrollHeight = window.innerHeight * 3, enabled = true } = options;

  // Clamp value between min and max
  const clamp = (value: number, min: number, max: number) =>
    Math.max(min, Math.min(max, value));

  useEffect(() => {
    if (!enabled) return;

    const handleScroll = () => {
      // Calculate scroll progress as 0–1
      // document.documentElement.scrollHeight includes the artificially set height
      const maxScroll = document.documentElement.scrollHeight - window.innerHeight;
      scrollProgress.current = clamp(window.scrollY / maxScroll, 0, 1);
    };

    window.addEventListener('scroll', handleScroll, { passive: true });
    return () => window.removeEventListener('scroll', handleScroll);
  }, [enabled]);

  useFrame(() => {
    if (!enabled || waypoints.length === 0) return;

    const progress = scrollProgress.current;

    // Map scroll progress to a position along the waypoint path
    // If 3 waypoints and progress = 0.5, we're halfway between waypoint 1 and 2
    const totalSegments = waypoints.length - 1;
    const segmentIndex = progress * totalSegments;
    const currentSegment = Math.floor(segmentIndex);
    const segmentProgress = segmentIndex - currentSegment;

    // Clamp to valid range
    const fromIndex = clamp(currentSegment, 0, waypoints.length - 1);
    const toIndex = clamp(currentSegment + 1, 0, waypoints.length - 1);

    const fromWaypoint = waypoints[fromIndex];
    const toWaypoint = waypoints[toIndex];

    // Lerp camera position and target between waypoints
    const nextPosition = new Vector3().lerpVectors(
      fromWaypoint.position,
      toWaypoint.position,
      segmentProgress
    );
    const nextTarget = new Vector3().lerpVectors(
      fromWaypoint.target,
      toWaypoint.target,
      segmentProgress
    );

    // Smoothly move camera toward the calculated position
    camera.position.lerp(nextPosition, 0.1);

    // Smoothly lerp FOV
    const nextFov = fromWaypoint.fov + (toWaypoint.fov - fromWaypoint.fov) * segmentProgress;
    camera.fov += (nextFov - camera.fov) * 0.05;
    camera.updateProjectionMatrix();

    // Update camera look-at
    camera.lookAt(nextTarget);
  });
}

Hook Explanation

Scroll listener: We add a passive scroll event listener (passive: true means the listener won't block scrolling). We calculate scrollProgress as a ratio of window.scrollY to the maximum scroll distance. This gives us a value from 0 to 1.

Waypoint interpolation: In useFrame, we map the scroll progress (0–1) to a position along the waypoint path. If we have 3 waypoints (overview, entrance, terrace), we divide the 0–1 range into 2 segments: 0–0.33 transitions from overview to entrance, 0.33–0.67 from entrance to terrace, and 0.67–1 stays at terrace. We use lerpVectors to smoothly interpolate position, target, and FOV between waypoints.

Smooth camera movement: We don't snap the camera directly to the calculated position. Instead, we lerp toward it with factor 0.1, so the camera smoothly "catches up" to the scroll-driven waypoint. This prevents the camera from snapping and feels more organic.

        <ContactShadows position={[0, -0.01, 0]} opacity={0.4} scale={50} blur={2} />

        <fog attach="fog" args={['#e8e0d0', 50, 150]} />

        <HotspotManager />

        {/* OrbitControls: disabled in scroll mode to avoid conflicts */}
        {cameraMode === 'orbit' && <OrbitControls />}
      </Suspense>
    </Canvas>
  );
}

Integration Explanation

Mode switching: We read cameraMode from the Zustand store. If it's 'scroll', we enable scroll-driven camera and disable OrbitControls. If it's 'orbit', we do the opposite. This prevents conflicts where both systems try to move the camera.

Document height: When cameraMode is 'scroll', we set document.body.style.height = '300vh'. This artificially extends the scroll area to 3 times the viewport height, giving the user plenty of scroll distance to traverse the three waypoints. On unmount, we reset height to 'auto'.

Pointer events: We set pointerEvents = 'none' on the Canvas when in scroll mode. This prevents clicks on the canvas from being processed (which might trigger hotspots or other 3D interaction). The user is scrolling, not clicking, so we disable interaction.

Waypoint array: We define three waypoints that match the camera positions from Chapter 11 (overview, entrance, terrace). As the user scrolls, the camera interpolates between these waypoints.

Handling Scroll Conflicts and Edge Cases

Problem: Scroll mode + hotspots. If you enable hotspots while in scroll mode, clicking a hotspot will interfere with scroll flow. Solution: conditionally disable hotspots or close any open hotspot when scroll mode is active.

// In HotspotManager, disable hotspot interaction in scroll mode:
import { useExperienceStore } from '../store/experienceStore';

export function HotspotManager() {
  const cameraMode = useExperienceStore((state) => state.cameraMode);
  const [activeHotspotId, setActiveHotspotId] = useState<string | null>(null);

  // Close hotspot when switching to scroll mode
  useEffect(() => {
    if (cameraMode === 'scroll') {
      setActiveHotspotId(null);
    }
  }, [cameraMode]);

  if (cameraMode === 'scroll') {
    // Don't render hotspots in scroll mode
    return null;
  }

  // ... rest of HotspotManager
}

Problem: Mobile scroll conflicts with canvas. Mobile browsers handle scroll differently. Solution: test on mobile and consider disabling scroll mode on touch devices (see Chapter 18).

🔑 Key Concept

Scroll-driven animation is powerful but requires careful orchestration. The camera must be synchronized with the user's scroll position, not fighting against it. If the user scrolls fast, the camera should follow fast (but smoothly). If they scroll slow, the camera should follow slow. The lerp factor (0.1 in our code) controls how quickly the camera "chases" the scroll-driven target — adjust it based on how responsive you want the camera to feel.

⚠ Common Mistake

Enabling both OrbitControls and useScrollCamera simultaneously. They will conflict, and the camera will jitter or behave unpredictably. Always conditionally enable one or the other based on cameraMode. Similarly, don't enable hotspot clicks in scroll mode — they interfere with scroll flow.

Mobile Simplification and Fallbacks

What You Will Build

A mobile detection and graceful degradation system. On desktop, users see the full WebGL canvas with complex lighting, shadows, and animations. On mobile, users see a high-quality static image. You'll learn that a perfect experience on 95% of devices (desktop + high-end mobile) is better than a janky experience on 100% of devices.

The Mobile Reality

WebGL is computationally expensive. A scene with directional shadows, point lights, HDRI environment, and scroll animations can run at 60fps on a MacBook Pro but at 15fps on a mid-range Android phone. A user on a 3G connection may wait 5+ seconds for your HDRI file to download. A touch interaction on a mobile device has different expectations than mouse interaction on desktop.

The solution is not to make the scene simpler for everyone. It's to detect the device capability and deliver an appropriate experience: flagship desktop = full interactive 3D, mid-range mobile = static image or simplified WebGL, low-end mobile = static image only.

Detection Utility

// src/utils/deviceDetection.ts
export function isMobileDevice(): boolean {
  // Check navigator.maxTouchPoints (all touch devices have this)
  if (navigator.maxTouchPoints > 0) {
    return true;
  }

  // Fallback: check user agent for mobile patterns
  const userAgent = navigator.userAgent.toLowerCase();
  return /mobile|android|iphone|ipad|ipod|blackberry|windows phone/.test(userAgent);
}

export function isTablet(): boolean {
  return /ipad|android(?!.*mobi)/i.test(navigator.userAgent.toLowerCase());
}

export function isHighEndMobile(): boolean {
  // Simple heuristic: if the device has 4+ cores and 4GB+ RAM, it's capable
  // navigator.hardwareConcurrency is the core count (if available)
  // This is a rough estimate; in reality, you'd measure performance at runtime
  return navigator.hardwareConcurrency >= 4;
}

export function getTargetDPR(): number {
  // Desktop: 2x DPR (high quality)
  // Mobile: 1x DPR (lower quality to save performance)
  return isMobileDevice() ? 1 : 2;
}

export function getCanvasResolution(): number {
  // Mobile: lower resolution for better performance
  // Desktop: full resolution

    return window.innerHeight < 800 ? 0.5 : 0.75; // 50% or 75% resolution

  return 1; // 100% resolution

    <Canvas
      shadows="soft"

        antialias: true,
        alpha: true,
        preserveDrawingBuffer: false,
        powerPreference: 'high-performance',

      camera={{ position: [25, 18, 35], fov: 45 }}
    >
      {/* Scene content */}
    </Canvas>

}

dpr (device pixel ratio): dpr={[1, dpr]} is the only knob you need. R3F clamps the renderer pixel ratio between those bounds. There is no pixelRatio field on WebGLRenderer; do not pass one inside gl={{ ... }}. Lowering the upper bound from 2 to 1 quarters the pixel count, with little perceived loss on a small mobile screen.

gl props: powerPreference: 'high-performance' requests the discrete GPU when available. preserveDrawingBuffer: false (the default) lets WebGL optimize memory.

Performance Optimization

What You Will Build

A performance monitoring and optimization workflow. You'll add performance instrumentation to PropertyExperience, identify bottlenecks (draw calls, memory), and implement targeted optimizations: geometry instancing, texture compression, LOD (Level of Detail), and draw call reduction.

Measure First, Optimize Second

Never optimize blindly. Always measure first. Use browser tools to understand where the performance cost is: is it the GPU (rendering), CPU (logic), or memory (loading)? Is it the HDRI file download, the GLB model, or the lighting calculations? Once you know the bottleneck, optimize it.

PerformanceMonitor shows FPS, memory usage, and GPU metrics in the top-left corner. Watch for these red flags:

  • FPS < 30: The scene is too expensive for the device. Disable shadows, HDRI, or reduce geometry.
  • Memory > 500MB: You're loading too much data. Compress textures or models.
  • Draw calls > 100: You're rendering too many separate meshes. Use instancing or merging.

Browser DevTools Performance Profiling

Open Chrome DevTools (F12), go to the "Performance" tab, click Record, interact with your scene for 10 seconds, then stop. You'll see a timeline showing:

  • Scripting (JavaScript execution time)
  • Rendering (WebGL draw calls)
  • Painting (DOM updates)

If Rendering is the bottleneck, optimize geometry and draw calls. If Scripting is high, optimize your update loop (avoid creating objects in useFrame, cache calculations).

Draw Calls and Instancing

Each mesh in your scene is one draw call. 10 meshes = 10 draw calls. 100 meshes = 100 draw calls. If you have many identical objects (e.g., a row of trees, repeated columns), use InstancedMesh to render them as a single draw call:

// src/experience/InstancedTrees.tsx
import { useRef, useEffect } from 'react';
import { InstancedMesh, Object3D } from 'three';

interface InstancedTreesProps {
  count: number;   // Number of trees
  spacing: number; // Distance between trees
}

export function InstancedTrees({ count, spacing }: InstancedTreesProps) {
  const meshRef = useRef<InstancedMesh>(null);

  useEffect(() => {
    if (!meshRef.current) return;

    // Reusable dummy Object3D so we don't allocate per instance.
    const dummy = new Object3D();
    for (let i = 0; i < count; i++) {
      dummy.position.set(i * spacing, 0, 0);
      dummy.updateMatrix();
      meshRef.current.setMatrixAt(i, dummy.matrix);
    }
    meshRef.current.instanceMatrix.needsUpdate = true;
  }, [count, spacing]);

  // Use JSX intrinsics for geometry and material — they live for the
  // component's lifetime and dispose automatically on unmount.
  return (
    <instancedMesh ref={meshRef} args={[undefined, undefined, count]}>
      <coneGeometry args={[0.5, 3, 8]} />
      <meshStandardMaterial color="#2d5016" />
    </instancedMesh>
  );
}

Result: 100 identical trees rendered as 1 draw call instead of 100. This is a 100x reduction in draw call overhead.

Texture Compression

Large texture files slow down loading and increase GPU memory. Use compressed texture formats (KTX2/Basis) for web:

// Before optimization:
// textureUrl: '/textures/marble.jpg' (2MB, uncompressed)

// After optimization:
// textureUrl: '/textures/marble.ktx2' (200KB, compressed)

// In your model or material:
<mesh>
  <meshStandardMaterial
    map={useTexture('/textures/marble.ktx2')} // Compressed texture
  />
</mesh>

To create compressed textures, use tools like Basis Universal or KTX2 converters (e.g., gltf-transform). The file size reduction is typically 5–10x with no visible quality loss.

GLB Model Compression

If your Aurelia model is a large GLB file, compress it using Draco compression:

// Terminal — compress a GLB with Draco using gltf-transform:
//   npm install -g @gltf-transform/cli
//   gltf-transform optimize aurelia.glb aurelia-optimized.glb --compress draco

// In your component, load the optimized model with the useGLTF hook
// (it is a hook, not a JSX component) and pass the Draco decoder path:
function AureliaModel() {
  const { scene } = useGLTF(
    '/models/aurelia-optimized.glb',
    'https://www.gstatic.com/draco/versioned/decoders/1.5.6/'
  );
  return <primitive object={scene} />;
}

Draco compression reduces GLB file size by 50–80% with minimal quality loss. A 10MB model becomes 1–2MB.

// In R3F, drei's useGLTF handles LOD automatically if the model defines it

Avoiding Common Performance Pitfalls

// Bad: Creating objects inside useFrame
useFrame(() => {
  const vec = new Vector3(); // Creates a new Vector3 every frame — memory leak!
  camera.position.lerp(vec, 0.05);
});

// Good: Define objects outside useFrame
const vec = useRef(new Vector3());
useFrame(() => {
  // Reuse the cached Vector3
  camera.position.lerp(vec.current, 0.05);
});

// Bad: useThree inside render loop
useFrame(() => {
  const { camera } = useThree(); // Calling hooks inside useFrame — error!
});

// Good: useThree at component level
const { camera } = useThree();
useFrame(() => {
  camera.position.lerp(...);
});

21.1 Model Sourcing

Where does the production model come from? The options, in order of increasing quality and effort:

Sketchfab: The largest library of 3D models on the web. Free models use CC0 or CC-BY licences; always check the licence before using in a commercial project. Search for "apartment building," "modernist house," or "luxury residence." Download as GLB. Polygon counts vary wildly — check that the model is under 500K triangles before committing.

BIM export: Revit, ArchiCAD, and Vectorworks all export to IFC. Convert IFC to GLB via BlenderBIM or IfcOpenShell + Blender. This gives you an accurate architectural model, but raw BIM exports are heavy (often 50–200MB) and need significant cleanup. Use for final client deliverables only, not for early development.

Purchased assets: TurboSquid, CGTrader, and Fab (Epic's marketplace) sell studio-quality architectural models at prices ranging from $30 to $800. Most are provided in FBX or OBJ format — convert to GLB via Blender.

Custom Blender modelling: Full control, maximum effort. For concept projects and portfolio pieces, modelling the Aurelia building from reference drawings in Blender is a valuable skill that makes you self-sufficient on client projects.

21.2 The Blender Cleanup Checklist

Every model — regardless of source — needs a cleanup pass in Blender before export. Raw exports are reliably broken in at least one of these ways:

  1. Apply all transforms. Select all objects → Object → Apply → All Transforms. Un-applied transforms cause the GLB exporter to produce incorrectly scaled or rotated geometry.
  2. Merge duplicate vertices. Edit Mode → Select All → Mesh → Merge by Distance (threshold: 0.001m). Eliminates invisible seams that split normals and cause shading artefacts.
  3. Remove non-rendering geometry. Delete interior faces, hidden objects, and any geometry the camera never sees. On an exterior building model, delete the interior floor slabs and wall cavity geometry — they add triangles but never render.
  4. Check and fix normals. Edit Mode → Overlay → Face Orientation. Red faces have inverted normals; blue faces are correct. Select red faces → Mesh → Normals → Flip.
  5. Limit material count. A single mesh should not have more than 8 materials. More than this causes multiple draw calls per object. Merge similar materials (all concrete surfaces share one material, all glass surfaces share another).
  6. Apply all modifiers. Any unapplied modifier (Subdivision Surface, Mirror, Array) will be exported as-is in some exporters and ignored in others. Apply all modifiers before export.
  7. Verify scale. The building should be life-scale: 1 unit = 1 metre. Check the bounding box (Object Properties → Dimensions). A 10-storey building should be approximately 30m tall.

21.5 Draco Compression

Draco geometry compression reduces GLB file size by 60–80% with no visible quality loss. It is supported natively by useGLTF via the DRACOLoader. Run it on the exported GLB using gltf-transform:

# Install once globally:
npm install -g @gltf-transform/cli

# Compress with Draco (level 6 is a good balance of size and quality):
gltf-transform optimize aurelia-lobby.glb aurelia-lobby-v1.glb \
  --compress draco \
  --draco.quantizationVolume mesh \
  --draco.method edgebreaker

# Verify output:
gltf-transform inspect aurelia-lobby-v1.glb
# Check: file size, triangle count, texture sizes, material count

Target: the compressed GLB should be 2–8MB for a hero architectural scene. If it is above 10MB, the model needs further polygon reduction before compression.

21.6 Texture Workflow

Source textures from Poly Haven at 2K resolution. Process them before use:

# Install sharp CLI for image processing:
npm install -g sharp-cli

# Convert albedo/diffuse maps to WebP (smallest file, best quality ratio):
sharp marble-01-albedo-2k.jpg -o marble-01-albedo-2k.webp --quality 85

# Normal maps: use lossless WebP or PNG (lossy compression breaks normals):
sharp marble-01-normal-2k.jpg -o marble-01-normal-2k.webp --lossless

# Resize to 1K for secondary surfaces:
sharp concrete-01-albedo-2k.webp -o concrete-01-albedo-1k.webp --resize 1024

For production texture delivery, KTX2/BasisU is the most performant format: it is GPU-decompressed directly, saving 4–6× the VRAM compared to WebP (which is CPU-decompressed first). Use drei's KTX2Loader:

npm install -g @ktx/ktx2

# Convert WebP to KTX2:
toktx --t2 --encode uastc --uastc_rdo_l 4 \
  marble-01-albedo-2k.ktx2 marble-01-albedo-2k.webp

# Load in R3F with KTX2Loader:
import { useKTX2 } from '@react-three/drei';
const texture = useKTX2('/textures/marble-01-albedo-2k.ktx2');

21.8 Naming Conventions

Consistent naming enables versioning and CDN cache control:

  • Models: aurelia-lobby-v1.glb, aurelia-lobby-lod-v1.glb
  • Textures: marble-01-albedo-2k.webp, concrete-02-normal-1k.webp

22.1 How Framer Code Components Work

A Framer code component is a TypeScript file that exports a default React component plus an optional addPropertyControls call. Framer imports it and renders it on the canvas. Every prop defined in addPropertyControls appears as a visual control in the properties panel — a color picker, a dropdown, a text field, an image picker, a boolean toggle.

The component receives those props exactly as a React component receives props. It is standard React. The only Framer-specific imports are addPropertyControls and ControlType from 'framer'.

22.2 Canvas Sizing in Framer

This is the most common mistake when building Framer code components that contain WebGL. Never set explicit pixel dimensions on the Canvas wrapper. In Framer, the component frame (set by the designer on the canvas) is the container. The component must fill it:

// ✅ Correct — fills the Framer frame
export default function AureliaExperience() {

23.5 Scroll Conflict Resolution

When a Webflow site has IX2 (Interactions 2.0) animations on the same page as an R3F embed in scroll mode, the two scroll listeners conflict. The symptoms: IX2 animations jitter, the R3F camera moves erratically, or both systems compete for the scroll position.

The production solution for Webflow: do not use scroll-hijacking in the R3F embed. Instead, drive the camera narrative from canvas-internal touch/drag interaction:

// Instead of scroll-hijacking, use internal pointer drag to advance the narrative:
// User drags right = advance to next waypoint
// User drags left = go to previous waypoint
// Webflow's page scroll works completely independently

useEffect(() => {
  let startX = 0;
  const container = canvasRef.current;

  const onPointerDown = (e: PointerEvent) => { startX = e.clientX; };
  const onPointerUp = (e: PointerEvent) => {
    const delta = e.clientX - startX;
    if (Math.abs(delta) > 50) {
      delta < 0 ? advanceNarrative() : retreatNarrative();
    }
  };

  container?.addEventListener('pointerdown', onPointerDown);
  container?.addEventListener('pointerup', onPointerUp);
  return () => {
    container?.removeEventListener('pointerdown', onPointerDown);
    container?.removeEventListener('pointerup', onPointerUp);
  };
}, []);
⚠ Webflow Cautions

Container height: Always set explicit height on the container div. If you forget, the Canvas renders at 0px height and nothing is visible. Webflow's preview mode may show a default height; the published site will not.

Script placement: Place the embed script after the target div, or in Webflow's Page Settings → Footer Code. Never place it in the <head> — the target div does not exist yet when the script runs.

CSP and CDN: Webflow's Content Security Policy may block scripts from CDN domains it does not recognize. Test your CDN URL in Webflow preview before launch. If blocked, use a Webflow-approved CDN or self-host the bundle on your own domain.

Webflow global CSS: Webflow's reset styles (* selector, body margin, etc.) cascade into <Html> overlay content. Scope all hotspot panel CSS with a high-specificity class name (.aurelia-hotspot-panel) and declare all layout properties explicitly.

Double initialization: If the embed block accidentally appears twice (or if the page is in Webflow's Editor mode), the script runs twice. The double-init guard in embed-entry.ts (if (container.dataset.initialized) return) prevents two React roots from mounting into the same container.
